# TwitchMetrics Architecture

## 1. Project Overview

TwitchMetrics is a multi-platform creator analytics dashboard that tracks and visualizes growth metrics across Twitch, YouTube, Instagram, TikTok, X, and Kick.

**Three user types:**

- **Creators** — view their own analytics, claim profiles, connect OAuth accounts
- **Talent Managers** — monitor multiple creators, export data
- **Brands** — discover creators, evaluate partnerships

**Cold-start strategy:** Pre-populated creator profiles (unclaimed) from public API data. Creators can claim profiles via OAuth verification or challenge-based methods.

## 2. Monorepo Layout

```
apps/
  web/              Next.js 15 app (App Router, Server Components, tRPC)
packages/
  database/         Prisma schema, client, migrations
  config/           Shared ESLint + TypeScript configs
  ui/               Shared UI components (future)
```

## 3. Canonical Data Flow

```
Platform API → PlatformAdapter → Inngest Worker → MetricSnapshot (DB)
MetricSnapshot → Rollup Worker → CreatorGrowthRollup (DB, future)
CreatorGrowthRollup → tRPC Router → Server Components → Browser
```

- **Ingestion:** Inngest cron functions call platform adapters, write `MetricSnapshot` rows
- **Rollup:** Separate workers aggregate snapshots into growth rollups (planned)
- **Serving:** tRPC routers query rollups, Server Components render initial HTML

## 4. Core Abstractions

### 4.1 Platform Adapters

**Contract:** `src/server/adapters/types.ts`

Every platform integration implements the `PlatformAdapter` interface:

- `fetchProfile(username)` — resolve a username to profile data
- `fetchSnapshot(userId, options)` — fetch current metrics for a user
- `search(query, limit)` — search for creators on the platform
- `fetchTopGames?(limit)` — optional, fetch top games/categories

Adapters return normalized data types (`CreatorProfileData`, `CreatorSnapshotData`). They throw `AdapterError` with typed error codes (`rate_limited`, `auth_expired`, `not_found`, `api_error`, `network_error`).

### 4.2 Metric Registry

**Source of truth:** `src/lib/constants/metrics.ts`

The `METRICS` const object defines all 18 tracked metrics with:

- `key` — unique identifier (matches const key)
- `label` / `description` — human-readable text
- `format` — `number`, `percentage`, `duration_seconds`, `currency_usd`
- `platforms` — which platforms provide this metric
- `requiresOAuth` — whether the metric needs creator consent
- `snapshotField` — maps to `MetricSnapshot` Prisma field or `"extendedMetrics"` JSON

`MetricKey` is the derived union type of all metric keys. Use `getMetricsForPlatform(platform)` to filter.

### 4.3 Snapshot Tiers

**Source of truth:** `src/lib/constants/tiers.ts`

Creators are assigned a `SnapshotTier` based on follower count:

| Tier  | Threshold  | Cadence               | Cron          |
| ----- | ---------- | --------------------- | ------------- |
| tier1 | >= 100K    | Every 6 hours         | `0 */6 * * *` |
| tier2 | >= 10K     | Daily at 2am UTC      | `0 2 * * *`   |
| tier3 | All others | Weekly Sunday 3am UTC | `0 3 * * 0`   |

**Promotion:** Automatic when followers exceed next tier threshold.
**Demotion:** Requires 3 consecutive snapshots below demotion threshold (sticky).

### 4.4 API Envelope

All tRPC procedures return a standard envelope:

```typescript
{
  data: T          // the result payload
  error: {         // null on success
    code: string
    message: string
  } | null
  meta: {          // optional metadata
    cursor?: string
    total?: number
  }
}
```

## 5. Server/Client Boundaries

- `src/server/` — server-only code (adapters, tRPC routers, auth). NEVER import from Client Components.
- `src/lib/constants/` — pure constants with no side effects. Safe for both server and client.
- **Public pages** (e.g., creator profiles, leaderboards) — Server Components with direct DB reads for SEO and performance.
- **Dashboard pages** (authenticated) — tRPC client calls for real-time data.

## 6. Database Conventions

- **Entity tables** (`CreatorProfile`, `PlatformAccount`, `User`) — mutable, represent current state
- **Snapshot tables** (`MetricSnapshot`, `GameViewerSnapshot`) — append-only time series
- All dates stored in **UTC**, formatted as **ISO 8601**
- `BigInt` for follower/view counts (platforms exceed 32-bit int range)
- OAuth tokens encrypted via **AES-256-GCM** (`src/lib/encryption.ts`)
- UUIDs generated by Postgres (`gen_random_uuid()`)

## 7. Architecture Contract

These rules are normative. All code MUST comply.

1. **Adapter-only integration.** All platform API calls MUST go through a `PlatformAdapter` implementation. No direct API calls from routes, components, or workers.

2. **Entity/snapshot separation.** Entity tables (current state) and snapshot tables (time series) MUST NOT be conflated. Snapshots are append-only.

3. **Rollup-based growth metrics.** Growth metrics (day-over-day, week-over-week) MUST come from `creator_growth_rollups` (future). NEVER compute growth on-the-fly from raw snapshots in user-facing queries.

4. **Metric keys from registry.** All metric references MUST use `MetricKey` from `src/lib/constants/metrics.ts`. No ad-hoc string literals for metric names.

5. **Platform config from constants.** Platform metadata MUST come from `src/lib/constants/platforms.ts`. No hardcoded platform strings outside this file.

6. **Tier rules from constants.** Snapshot cadence and tier thresholds MUST come from `src/lib/constants/tiers.ts`. Inngest cron expressions should reference `TIER_CONFIG`.

7. **Server Components for public pages.** Public-facing pages MUST use Server Components for initial data load to ensure SEO and fast TTFB.

8. **ECharts for charts.** All chart visualizations MUST use Apache ECharts (`echarts-for-react`). No mixing chart libraries.

9. **API envelope.** All tRPC procedures MUST return `{ data, error, meta }` envelope shape.

10. **Worker responsibility separation.** Snapshot workers only write snapshots. Rollup workers only write rollups. No worker should do both.

11. **Auth identity separation.** Auth identity = email/magic link via NextAuth. Social OAuth connections = `ConnectedAccount` / `PlatformAccount` for data access, NOT for login.

12. **Claiming flow.** Profile claiming = linking a `ConnectedAccount` to a `CreatorProfile` via `platform_user_id` match. Uses `ClaimRequest` for verification.

13. **UTC everywhere.** All dates MUST be stored in UTC. Frontend converts to local timezone for display only.

14. **Server-only boundary.** Server-only code lives in `src/server/`. Client Components MUST NOT import from `src/server/`.

15. **Strict TypeScript.** TypeScript strict mode enabled. Avoid `any` — use `unknown` with type guards instead.

16. **Path alias.** `@/` path alias is rooted at `src/`. Use `@/lib/...`, `@/server/...`, `@/components/...`.

17. **No schema invention.** Do NOT add Prisma schema fields that conflict with the roadmap. If a field is needed, check existing schema first.

18. **Ask, don't guess.** When requirements are unclear, ask for clarification rather than making assumptions.

## 8. Environment Variables

See `.env.local` for the complete list of required environment variables. All variables listed in `turbo.json` `globalEnv` must be set.

Key variable groups:

- **Database:** `DATABASE_URL`, `DIRECT_URL`
- **Redis:** `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`
- **Auth:** `NEXTAUTH_URL`, `NEXTAUTH_SECRET`
- **OAuth providers:** `TWITCH_CLIENT_ID/SECRET`, `GOOGLE_CLIENT_ID/SECRET`, etc.
- **Encryption:** `ENCRYPTION_KEY` (64-char hex string)
- **Inngest:** `INNGEST_EVENT_KEY`, `INNGEST_SIGNING_KEY`
- **Storage:** `R2_*` variables for Cloudflare R2
